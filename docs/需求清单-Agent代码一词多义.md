# 需求清单：把「词义星海」改造成「代码一词多义 + Agent 判读」

> 基于《调研结果-Agent代码一词多义.md》的推荐：路径①（本地 CodeBERT 主体）加路径②（Agent 判读叠加）。本文列出能复用什么、要改什么、要你提供什么、风险与里程碑，供决策。

---

## 一、目标方案一句话

主体用本地 CodeBERT 取代码标识符的上下文向量，画成代码语义星海，做一词多义可视化（逐层、义项轴、最近邻、案例全保留）。叠加一个 Agent 判读面板，用 chat API 现场判断某符号的含义并高亮它注入的上下文。

## 二、能直接复用的部分

这一节先说好消息：大半工程不用动。

- 渲染：星海、辉光、相机、交互、探针点、文字标签，全部不动。
- 可视化逻辑：逐层可分度曲线（A）、义项轴读数（B）、最近邻簇（C）、消歧置信（D）、案例库，逻辑照搬，只换喂进去的数据。
- 管线骨架：`export_semantic_galaxy.py` 的「取向量 → PCA/UMAP → 星海加案例」结构照搬。
- 后端骨架：`/api/locate` 的「输入 → 取向量 → 投影加最近邻」结构照搬。
- 验证与自测：`diagnose_polysemy.py` 和 headless 流程照用。

## 三、需要新增或改造的部分

这一节是真正要做的活，按模块列。

### 模型与数据

- 换模型：GPT-2 换成 CodeBERT（`microsoft/codebert-base`，本地，HuggingFace），取 token 上下文向量、逐层照旧。
- 标识符抽取：从代码里取标识符 token（不再是英文词），要处理 CodeBERT 的 subword 切分，最好按作用域或文件归类。
- 底图语料：采一批常见代码标识符（来自开源代码或标准库）替换现在的 5000 英文词。
- 案例：精选代码多义符号（`run`、`pool`、`key`、`cell` 等），每个准备两三个义项、各自的代码例句，预算义项落点、最近邻、逐层轨迹、可分度。
- 选层：在代码上重跑诊断，确认哪一层对代码义项最可分（不一定是第 9 层），沿用「先验证再开工」。

### 后端

- 改 `/api/locate`：输入从一句话改成一段代码加目标符号，用 CodeBERT 取上下文向量，再投影加最近邻。
- 新增 `/api/agent`（路径②）：输入代码加符号，调 chat API，返回结构化结果 `{判定义项, 依据或注入的代码行, 置信}`。

### 前端

- 输入框改成贴代码：多行、等宽字体、可指定目标符号（点选或输入）。
- 案例库换成代码多义符号。
- 新增 Agent 判读面板：显示 Agent 的判断结果，并高亮它注入的代码行。
- 文案与命名改成「代码词义星海」。

### Agent 层（路径②）

- 接入 chat API（chat completions 形态），设计 prompt：判断符号义项并列出依据上下文。
- 解析结构化输出（JSON）。
- 失败兜底：API 不可用时退回纯 CodeBERT 展示，保证演示不挂。

## 四、需要你提供的

- chat API：endpoint、key、模型名、可用额度。要的是 chat 接口，不是 embedding 接口（演示级少量调用即可）。
- 是否指定某个代码库或语言作为底图语料，默认用 Python 标准库。
- 算力：CodeBERT 本地 CPU 可跑，和现在的 GPT-2 同级，不需要额外硬件。

## 五、风险与取舍

这一节把可能踩的坑先摆出来。

- CodeBERT 的逐层义项可分度未必有 GPT-2 那么漂亮，要先用诊断脚本验证，不成立就换层或换 GraphCodeBERT。
- 代码标识符的命名风格杂，底图聚类可能不如英文词干净。
- 路径②依赖 API 稳定，演示一定要有离线兜底。
- 嵌入 API 给不了 token 级和逐层，所以主体的嵌入必须走本地 CodeBERT。

## 六、工作量与里程碑

这一节给一个粗略排期，按依赖顺序。

1. 验证：CodeBERT 在代码多义符号上的逐层可分度（约半天）。
2. 管线：代码标识符星海加案例（约一天）。
3. 后端：`/api/locate` 代码版（约半天）。
4. 前端：贴代码输入、案例、命名（约半天）。
5. Agent 判读面板加 API 接入（约一天，路径②）。
6. 文档加 headless 自测（约半天）。

主体（第 1 到 4 步）做完就有一个能演示的「代码一词多义星海」；第 5 步是 Agent 亮点，可作为加分项后做。
