# 创新点

## 一、组合创新：编译器词法分类 + LLM 逐词语义

现有的代码教学工具在「怎么帮初学者理解代码」这件事上，通常走两条路线：一条是纯静态分析（语法高亮、AST 可视化），能告诉你这个词属于什么类型，但不解释它在这段代码里具体干什么；另一条是 AI 整段解释（ChatGPT、Copilot Chat），能讲清楚代码逻辑，但粒度停留在行或段，跳过了初学者最需要的「这个词是什么」。

本项目将两者在 **token 粒度**上组合：Pygments 提供编译器级别的词法分类（确定性的、快速的、多语言的），LLM 补充每个标识符在当前上下文中的具体语义（灵活的、上下文相关的）。这个组合在已有工具中没有先例。

与相关工具的对比：

| 工具 | 粒度 | 分类方式 | 语义解释 | 面向用户 |
|---|---|---|---|---|
| 语法高亮（VS Code 等） | token | 静态正则 / TextMate | 无 | 所有开发者 |
| AST Explorer | AST 节点 | 编译器解析 | 无（只展示语法树结构） | 有经验的开发者 |
| explainshell.com | 命令片段 | 手工标注的 man page | 静态文档摘录 | Shell 用户 |
| AI 代码解释（ChatGPT 等） | 行或段 | 无分类 | LLM 生成 | 所有用户 |
| **代码词法地图** | **token** | **Pygments 编译器级词法** | **LLM 逐词上下文语义** | **编程初学者** |

## 二、分级解释策略

系统对不同类型的 token 采用不同的解释策略，在速度和深度之间做了分层：

- 语言层面含义恒定的词（关键字、运算符、self/cls 等）用固定字典，**0 毫秒**返回。
- 能从上下文推断角色的词（def 后面的一定是函数名，class 后面的一定是类名）用规则引擎，同样**0 毫秒**返回，并在卡片中先行显示。
- 需要理解代码逻辑才能解释的标识符（用户定义的变量、函数调用等）交给 LLM，约 **1 秒**返回后替换快速提示。

这个分层的效果是：用户点击任何词都能立即看到有意义的信息，LLM 的调用量也大幅减少（一段典型代码中约 60% 的 token 由前两级覆盖）。

## 三、一词多义的可视化呈现

同一个标识符在不同行可以承担完全不同的角色。例如 `data` 可以是 `import json as data` 中的模块别名、`def data(data)` 中的函数名和参数名、`class data` 中的类名。传统语法高亮对这四处 `data` 的着色完全相同，无法体现差异。

本系统中，这四处 `data` 虽然同属「标识符」区域，但点击后获得的语义解释各不相同：LLM 根据所在行的上下文分别给出针对性的解释。地图上同名 token 的圆点大小也反映了出现频率，视觉上提示读者「这个词出现了很多次，可能在不同地方有不同含义」。

## 四、按需解释与并发执行

早期版本在分析时一次性解释所有 token，一段 20 行的代码需要等待约 50 秒。改为按需解释后，词法地图 0.1 秒即出，用户只为自己感兴趣的词付出等待成本。

多线程并发让用户快速连点多个标识符时，所有 LLM 请求同时发出。5 个并发请求的总耗时从串行的 3.2 秒降至 0.7 秒，与点击单个词的延迟几乎相同。

## 五、零门槛的交互设计

系统面向编程初学者，交互设计围绕「降低认知负担」展开：

- 分级样例从最简单的三行代码（`x = 10`）开始，逐步过渡到函数定义和一词多义场景，让初学者有明确的探索路径。
- 三视图联动（代码面板、地图、语义卡）让用户无论从哪个入口点击，都能同时看到这个词在代码中的位置、在词法分类中的归属、以及在当前上下文中的含义。
- 整个前端无框架依赖，从 `http://127.0.0.1:5000` 打开即用，不需要安装任何工具链。
