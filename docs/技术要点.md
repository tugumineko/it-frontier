# 技术要点

## 整体架构

系统分为三层：Pygments 词法引擎、ECNU LLM 语义引擎、浏览器前端交互层。后端是一个 Flask 应用，托管前端静态文件并提供两个 API 端点。

```
用户代码 → /api/analyze（Pygments 词法分析）→ 四类 token + 频率权重 → 前端渲染 SVG 地图
                                                                    ↓
                                                  点击标识符 → /api/explain（ecnu-max 单词解释）→ 语义卡片
```

## 词法分析层

Pygments 是 Python 生态中使用最广泛的词法分析库，支持数百种编程语言。系统调用 `lexer.get_tokens_unprocessed()` 获取每个 token 的类型标记（如 `Token.Keyword`、`Token.Name`、`Token.Operator`），再通过 `categorize()` 函数映射到四大类。

这一层的设计要点：

- **确定性分类**。Pygments 的分类结果与编译器词法阶段一致，不存在概率性误判。
- **字符串片段合并**。Pygments 会把 `"hello"` 拆成引号和内容两个 token，系统在 `lex_code()` 中将相邻的 String 类型 token 合并为一个完整字面量。
- **行列定位**。每个 token 记录所在行号和列偏移，用于前端代码面板的精确高亮。

## 语义解释层

语义解释分为三个层级，按优先级依次尝试：

**第一级：固定字典**。关键字（`KW_HINT`，26 条）、运算符（`OP_HINT`，17 条）、常见标识符（`ID_HINT`，7 条，如 self、cls、\_\_init\_\_）使用硬编码解释。这些词在语言层面含义恒定，不需要消耗 LLM 资源，响应时间为 0。

**第二级：上下文快速提示**。`ctx_hint()` 函数检查当前 token 的前一个 token，推断角色。例如前一个 token 是 `def` 则当前词标注为「函数名」，前一个是 `class` 则标注为「类名」。这一层同样不调用 LLM，在 `/api/analyze` 阶段就完成，结果存储在 token 的 `quick` 字段中。

**第三级：LLM 深层解释**。对于没有固定含义的标识符，前端在用户点击时调用 `/api/explain`，向 ecnu-max 发送当前代码和目标标识符的位置，获得一句不超过 20 字的上下文相关解释。通过 `max_tokens: 80` 和 `temperature: 0` 参数控制生成长度和确定性，平均响应时间约 0.8 秒。

## 前端交互层

前端是纯原生 JavaScript，无框架依赖，由三个模块组成：

- **main.js**：SVG 渲染引擎。在 1000x680 的 viewBox 中绘制四区矩形、网格排布 token 圆点（半径按出现频率缩放）、绘制词法轨迹折线。
- **ui.js**：控制逻辑。分析按钮的防重入锁（`busy` 标志）、分级样例一键填充、数据导入导出。
- **style.css**：深色星空主题，CSS 变量定义四类颜色。

三视图联动机制：`selectTok(id)` 函数同时更新地图圆点选中态、代码面板高亮行、语义卡片内容。

## 并发与性能

| 环节 | 策略 | 效果 |
|---|---|---|
| 词法分析 | 纯 Pygments，不调 LLM | 0.1 秒出地图 |
| 固定字典和上下文提示 | 硬编码查表 | 0 毫秒 |
| LLM 解释 | 按需调用，点击才请求 | 避免批量等待 |
| 多个标识符同时点击 | Flask `threaded=True` 多线程 | 5 个并发 3.2 秒降至 0.7 秒 |
| 请求防抖 | `reqSeq` 序列号丢弃旧响应 | 防止画面来回切换 |
| 按钮防重入 | `busy` 标志禁用按钮 | 避免请求积压 |

## 依赖

后端仅依赖 `flask` 和 `pygments` 两个包，LLM 调用使用标准库 `urllib`，无 torch、transformers 等重型依赖。前端无任何第三方库。
